## JWT
JWT 인증방식은 json web token의 약자로 인증에 필요한 정보들을 암호화 시킨 토큰을 말한다.우리가 사용하는 인터넷은 http 프로토콜을 따라 서버와 클라이언트 간의 응답과 데이터 요청이 이루어진다. http는 '비연결성', '무상태성'이라는 특징을 가지고 있다. 즉, 클라이언트의 요청에 따라 서버에서 응답을 처리하게 되면 연결 상태를 끊어버린다. 따라서 우리가 웹페이지를 새로고침 할 때마다 서버에서 클라이언트의 정보를 식별할 수 없다면 우리는 계속해서 로그인을 해야하는 상황에 직면하게 된다. 그러한 불편함을 없애주는 기술이 바로 cookie와 session이다. JWT인증방식은 cookie/session 와 유사하게 JWT 토큰(Access Token)을 발행해주는 인증방식을 말한다.

## API
api는 일종의 소프트웨어 인터페이스로 **컴퓨터와 소프트웨어를 연결**해주는 역할을 한다. 즉 컴퓨터와 소프트웨어를 연결시켜준다는 개념에서 결국 api는 일반 사용자가 아닌 컴퓨터 프로그래머가 사용하게 고안되었다고 할 수 있다. 흔히 우리가 말하는 유저 인터페이스, UI란 사용자와 컴퓨터, 인터넷등의 사물을 연결시켜주는 매개체를 말한다. 대표적인 예가 핸드폰의 홈 버튼이다. 홈 버튼을 누름으로 써 우리는 핸드폰에게 무언가 요청을 할 수 있고, 핸드폰은 그 요청을 받을 수 있다. 단순한 예 이지만 api도 크게 다르지 않다. 단지 api는 프로그램과 프로그램을 이어주는 역할을 한다는 점에서 차이를 가질 뿐이다. 또한 api 문서를 통해 **주고 받을 데이터의 사용법과 규격을 지정**한다. 그리고 api는 **내부에서 어떤 기능에 의해 구현되는지 알 필요 없이 그저 원하는 정보만 가져다 사용하면 된다.**
```
API 예시
      $.ajax({
      type: "GET",
      url: "http://spartacodingclub.shop/sparta_api/seoulair", //URL 주소
      data: {},
      success: function(response){
      console.log(response)
      }
      })
```

## 알고리즘
이번주 목요일부터 프로그래머스에 있는 문제를 풀어보면서 구현력을 기르고 있는데 생각보다 쉽지 않다. Lv.1부터 하고 있는데 처음 문제를 보자마자 아무 생각이 안났다. 그냥 바로 문제를 풀려고 하니까 덜컥 겁을 먹었는데 이제는 이 문제를 어떤 방식으로 풀어야하는지, 문제가 원하는게 뭔지 문제인식를 조금씩 하고 있다. 하지만 필요한 함수를 검색해 나가면서 문제를 풀다보니 시간이 오래 걸린다. 많이 하다보면은 이제 척하면 척이지 않을까? 그리고 무엇보다도 프로젝트 만드는 것보다 재미있다. 풀었을 때 성취감도 있고, 내가 푼 방식과 다른 사람이 푼 방식을 비교해보면서 여러방식들을 알아갈 수 있어서 좋다. (한줄로 끝내는 사람들은 진짜 천재들이야ㅠㅅㅠ) 나는 왜 저런 방식으로 생각을 못해봤지? 할 수도 있지만 이번에 처음 접하는 알고리즘이기 때문에 이런 생각보다 더 많은 문제를 더 많은 방식으로 풀어보자라고 생각하게 된다. 문법도 중요하고 함수 사용도 중요하지만 결국은 **사고의 전환, 생각하는 과정**등이 더 중요하다고 생각한다.
